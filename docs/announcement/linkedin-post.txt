Your most expensive AI coding model is probably spending most of its tokens on work a cheaper model does just as well.

Reading files. Running tests. Writing boilerplate.

You're paying for a supreme court justice and asking it to do data entry.

I spent the last few weeks fixing that - and the fix wasn't a cheaper model or worse output. It was a harness: the structure around the model, not the model itself.

The pattern, in one line: one frontier model judges, a flat-rate model builds, and the repository - not the chat - is the memory.

The orchestration best practices that make it work:

- Separate planning context from execution context
- The repo is the memory, not the chat history
- Fresh context per task (no context rot)
- The worker never grades its own work - a different model from a different lab reviews it
- Freeze the acceptance criteria before any results exist
- Disagreement is mandatory: the builder must challenge the spec before writing code
- Spend reasoning budget only where it survives review

Measured splits like this cut frontier-model spend 58-74% - and quality goes UP, not down, because the same split that saves money is what removes self-grading and goalpost-moving.

I packaged it as two open-source Claude Code skills, MIT-licensed, ~30-second install. Full writeup and the design doc (with ~40 cited sources) linked below. Repo in the first comment.
