Fable is the best judgment model ever shipped.

It is also the most expensive way ever invented to run npm test.

$10 in, $50 out per million tokens. Simon Willison burned $110 in a single day of normal use. And if you watch where a coding session actually spends those tokens, most of them go to work that doesn't need Fable at all — reading files, running commands, writing boilerplate, retrying lint.

You're paying for a supreme court justice and making it do data entry.

The fix isn't a cheaper model. Cheaper models give you cheaper judgment, and judgment is the only thing you actually wanted Fable for.

The fix is a harness.

> Harnesses are where the leverage is now

This isn't a fringe take anymore. One team moved their coding agent from Top-30 to Top-5 on a leaderboard changing only the harness — same model. A study improved 15 different LLMs at coding in an afternoon by changing only the harness. AI Engineer Europe just ran its first dedicated Harness Engineering track.

The model decides the ceiling. The harness decides how much of the ceiling you can afford.

So here's the harness I run, packaged as two open-source Claude Code skills:

Fable is the architect. Judgment only. It never writes implementation code. It arbitrates disagreements, judges evidence, writes specs, makes kill/continue calls. Minutes per work block.

GPT-5.5 Codex is the builder. A fresh codex exec process per slice, xhigh reasoning, running unattended for hours on the flat-rate ChatGPT plan. It leads Terminal-Bench 2.0 at 82.7% — typing is the thing it's best at.

The repo is the memory. Both models are stateless between sessions. State lives in docs/HANDOFF.md, docs/gates/, and git — which is exactly why a few minutes of Fable is enough: it reads state instead of reconstructing it from a 200k-token chat history.

Fable tokens go only where judgment compounds. Everything else runs on a subscription you already have. Measured orchestrator/worker splits like this cut spend 58–74% versus running the top model end-to-end.

> The loop is:

1. Fable reads docs/HANDOFF.md and rules on every disagreement the builder raised: ACCEPT / REJECT / MODIFY, one line why. No deferrals.

2. Fable judges the last slice by running the acceptance-gate commands itself and reading the raw output. Builder claims are hearsay. Verdict per gate: PASS / FAIL / INVALID.

3. Fable specs the next one-PR slice and freezes its acceptance gates to docs/gates/ — committed BEFORE any code exists.

4. Up to 4 fresh Codex processes build for hours — in parallel, each in its own git worktree, on its own disjoint file set. Each must argue with the spec before writing code, citing real files — silent compliance is a failure. Each reports raw numbers only. No adjectives. No "promising."

5. Fable reviews each lane, commits it, and merges — builders physically can't commit (the sandbox protects .git), so nothing reaches a branch until the architect's checks pass. You read the handoff between blocks. You own kill/continue.

The whole integration is one command:

codex exec --sandbox workspace-write -m gpt-5.5 -c model_reasoning_effort="xhigh" "<the spec>"

> Why quality goes UP, not down

This is the part people get wrong about cost-cutting harnesses. The naive version — use the cheap model for everything, escalate when stuck — does trade quality for cost.

This one doesn't, because the token split IS the quality mechanism:

The builder never grades its own work. A model reviewing its own output has the blind spots that produced the output. Here the judge is a different model from a different lab. Cross-vendor review kills same-model sycophancy structurally — OpenAI pitches their own Claude bridge on exactly this point.

The gates can't move. Acceptance criteria written after results exist always pass. So gates freeze in their own commit before dispatch, and the architect's post-run check runs git diff on docs/gates/ — if the builder so much as touches a gate file, the slice auto-fails regardless of results. Not paranoia: in published benchmarks, 47–74% of agent self-improvement runs showed fake gains — agents rewriting grading scripts and mocking test output.

Disagreement is mandatory. Before writing code, the builder must challenge the spec, citing real files. Spec bugs get caught before they compound into a six-hour run in the wrong direction — which is, incidentally, the most expensive token burn there is.

Fresh context per slice. Every slice is a new Codex process. No context rot, no "dumb zone." If a run wrecks the repo: git reset to the freeze commit, re-dispatch. Code is cheap. Rescue prompting isn't.

> The one place I spend MORE per token

[insert image: effort-chart.png]

The builder runs xhigh reasoning even though it's ~2.2x the cost of high. Independent benchmarking shows the gap isn't test-pass rate — it's whether the output survives review: review-pass 38% to 69%, matching the human PR 69% to 88%. For unattended multi-hour runs, review-survival is exactly the thing worth paying for. Cheap tokens that need rework aren't cheap.

That's the whole philosophy in one chart: spend aggressively where it compounds, spend nothing where it doesn't.

> Same trick for research

The second skill, /architect-research, applies the identical split to discovery work. Six parallel Codex researchers fan out — latest academic papers (arXiv + citation snowballing), most popular repos (ranked by dependents, because ~4.5M GitHub stars are documented fakes), cutting-edge repos, design patterns mined from production-grade libraries, general web, and a second wave that tracks the field's named experts.

Fable does only the judgment: verifying every load-bearing claim against 2+ independent sources, adversarial falsification searches, and one decision-oriented report. Citations only from URLs fetched during the run — even search-grounded agents fabricate 3–13% of citations.

Gathering on the flat rate. Judgment on Fable. Same harness, different job.

> Install

git clone https://github.com/DanMcInerney/architect-loop

cd architect-loop && ./install.sh

npm i -g @openai/codex@latest

Then in any repo: /architect-research <idea> to explore, /architect to build. First run creates the handoff and gates from templates. Requirements: Claude Code on any paid plan, Codex CLI 0.133+ on a ChatGPT plan. No API keys.

> What it's not

Honest limits. The loop is overkill for small tasks — a one-file fix doesn't need an architect, and the skill says so itself. Long builder runs draw on your ChatGPT weekly quota (a ~6.5-hour run is roughly 20% of a $100-tier week). The research skill runs ~15x chat-level tokens — that's why it's a deliberate command, not a default. And parts of this harness will be obsoleted by better models; the design doc has a standing rule to delete scaffolding every model generation. Even the Claude Code team argues the secret sauce is the model — they're half right, which is why this harness is thin.
