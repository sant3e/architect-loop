Your most expensive AI coding model is probably spending most of its tokens on work a cheaper model does just as well.

Reading files. Running tests. Writing boilerplate. Retrying lint.

I spent the last few weeks fixing that - and the fix didn't involve a cheaper model or worse output. It involved a harness. Here's the pattern, the orchestration best practices behind it, and the open-source skills that implement it.

> The problem, stated plainly

Claude Fable is the best judgment model I've used. It's also $10 in / $50 out per million tokens - Simon Willison reported burning $110 in a single day of normal use.

But when you watch where a coding session actually spends those tokens, most of them go to mechanical work that has nothing to do with judgment. You're paying for a supreme court justice and asking it to do data entry.

The instinct is to switch to a cheaper model. That's the wrong move: cheaper models give you cheaper judgment, and judgment is the only thing you wanted the expensive model for in the first place.

The right move is to change the harness - the code and structure around the model, not the model itself.

> Why the harness is where the leverage is

This is now a well-supported claim, not a hunch:

- One team moved a coding agent from Top-30 to Top-5 on a public leaderboard by changing only the harness - same model.
- A research study improved 15 different LLMs at coding in a single afternoon by changing only the harness.
- AI Engineer Europe ran its first dedicated "Harness Engineering" track this year.

The model sets the ceiling. The harness decides how much of that ceiling you can actually afford.

> The system: two models, one repo

I packaged the harness as two open-source skills for Claude Code. The architecture splits one job across two models from two different labs:

THE ARCHITECT - Claude Fable. Judgment only. It never writes implementation code. It arbitrates disagreements, judges evidence, writes specs, and makes kill/continue calls. Minutes of work per block.

THE BUILDER - GPT-5.5 via the Codex CLI. A fresh process per task, running unattended for hours on a flat-rate ChatGPT plan. It currently leads Terminal-Bench 2.0 at 82.7% - hands-on building is the thing it's best at.

THE MEMORY - the repository itself. Both models are stateless between sessions. State lives in a handoff document, a frozen-gates folder, and git history. That's the key to why a few minutes of the expensive model is enough: it reads state instead of reconstructing it from a 200,000-token chat log.

The result: the expensive model's tokens go only where judgment compounds. Everything else runs on a subscription you already pay for. Measured orchestrator/worker splits like this cut spend 58-74% versus running the top model end to end.

> The orchestration best practices we're using

This is the part worth bookmarking. Every rule below is something the serious sources on agentic systems - Anthropic's harness-engineering research, the most-installed community skills, the academic literature on reward hacking - converged on independently. The skills enforce each one mechanically, not as a suggestion:

1. Separate planning context from execution context. The model that decides what to build is not carrying the 200k-token history of building it. (Anthropic context-engineering research.)

2. The repository is the memory, not the chat. Structured artifacts - a handoff doc, a gates folder, git - outlive any context window. Not in the handoff = it didn't happen.

3. Fresh context per task. Every unit of work is a clean process. No context rot, no degradation past the "dumb zone." If a run goes wrong, reset and re-dispatch - generation is cheap.

4. The worker never grades its own work. A model reviewing its own output shares the blind spots that produced it. Verification is done by a different context - and here, a different model from a different lab, which removes same-model sycophancy entirely.

5. Freeze the acceptance criteria before any results exist. Criteria written after the fact always pass. Gates are committed before dispatch, and a git diff check fails the task automatically if they're touched. In published benchmarks, 47-74% of agent self-improvement runs showed fake gains - agents rewriting their own grading scripts. This closes that door.

6. Disagreement is mandatory. Before writing code, the builder must challenge the spec, citing real files. Silent compliance is treated as a failure. This catches spec bugs before they compound into hours of work in the wrong direction.

7. Every delegation carries a full contract: objective, output format, tool guidance, and explicit boundaries. Vague delegation is the single biggest cause of duplicated and misdirected agent work.

8. Verify with evidence, not assertions. The architect runs the acceptance commands itself and reads the raw output. A worker's claim that "tests pass" is hearsay until the orchestrator sees the result.

9. Scale effort to where it compounds. Spend the expensive reasoning budget on the work that survives review; spend nothing extra on the routine. (More on this below.)

> Why quality goes up, not down

The naive version of cost-cutting - "use the cheap model for everything, escalate when stuck" - genuinely does trade quality for cost.

This pattern doesn't, because the same split that saves money is the thing that raises quality. Cross-vendor review catches what self-review can't. Frozen gates make goalpost-moving structurally impossible. Mandatory disagreement catches errors at the cheapest possible moment - before the work starts. The cost savings and the quality gains come from the same design decision.

> The one place this harness spends MORE

There's exactly one lever set to maximum: the builder runs at the highest reasoning effort, roughly 2.2x the cost of the next tier down.

Independent benchmarking shows why. At that tier the gain isn't raw test-pass rate - it's whether the output survives review. Review-pass climbs from 38% to 69%; semantic match to a human-written PR climbs from 69% to 88%. For unattended multi-hour runs, review-survival is precisely the thing worth paying for. Cheap tokens that need rework aren't cheap.

That's the whole philosophy in one decision: spend aggressively where it compounds, spend nothing where it doesn't.

> The same pattern, applied to research

The second skill applies the identical split to discovery work. Six parallel researchers fan out - academic papers, most-adopted repositories, emerging projects, design patterns from production-grade libraries, general web, and a second wave tracking named experts in the field. The expensive model does only the judgment: verifying each load-bearing claim against at least two independent sources, running adversarial checks, and writing one decision-oriented report. Gathering on the flat rate; judgment on the frontier model.

> Honest limitations

I'd rather you trust this than oversell it. The loop is overkill for small tasks - a one-file fix doesn't need an architect, and the skill says so itself. Long builder runs draw on a weekly quota. The research skill is token-heavy by design, which is why it's a deliberate command and not a default. And some of this scaffolding will be obsoleted by better models - the design doc has a standing rule to delete what each new model generation makes unnecessary.

> Try it

The repo is MIT-licensed and installs in about 30 seconds. Requirements are just Claude Code on any paid plan and the Codex CLI on a ChatGPT plan - no API keys.

Every rule cites its source; the full design document with ~40 citations is in the repo.
